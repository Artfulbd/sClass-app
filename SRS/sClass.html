2. Overall Description

2.1 Product Perspective

The virtual classroom system promotes a greater count of students and minus the communicating hazards and expenses. Also it monitors study activity.

2.2 Product Functions
  1. • Only Registered users can access the course.
  2. • Users must have valid User id and password to access the system.
  3. • In this system student can be provided study material like any PDF file from the instructor.
  4. • Instructor can upload the syllabus, lesson plan, session plan, class notes and question bank for the courses.
  5. • System provides 24x7 availability.
2.3 User Characteristics

Each user has a valid user name, password, User ID, which department he/she belongs, Grade, etc.

2.4 Operating Environment

System requirements- Hardware

  1. • Platform: Atleast Microsoft server 2012
  2. • Space: Minimum of 5 gigabytes
  3. • CPU: Atleast Dual Core 3.6MHz
  4. • RAM: Minimum of 2 gigabytes

System requirements- Software

  1. • Web Service: Apache server
  2. • Other services: IIS 6.0 or later, video streaming

User requirements- Hardware

  1. • CPU: atleast 1GHz
  2. • RAM: atleast 1gigabytes
  3. • Modules: Web camera (for PC) / Front camera for(android phone), video cards

User requirements- Softdware

  1. • Browser: Google Chrome or any.
  2. • Android version: 6.0 or higher (only for android phone)
2.5 Design and Implementation Constraints
  1. • Hardware Constrains: The system requires a database in order to store persistent data.
  2. • Software Constrains: The development of the system will be constrained by the ability of required software such as web servers, database, API and development tools.
2.6 User Documentation

Both the enrolled students and faculty will be provided printable user manuals as per their account type.

2.7 Assumptions and Dependencies
  1. • Current System and database will still be available for use and archive./li>
  2. • Database will be modified to meet “vClass’s” requirement.
  3. • Users should have an adequate computer, network connections and wapcam or front camera.

4.2.1 Signup

Actor: Student , Teacher

Brief overview: Any new user can create a new account using signup button.

Actor action System response
1. Any new actor goes to http://localhost/sClass/public/login.php 2. The System respons with the login page.(Fig:3.1.1a)
3. Because of having no account, actor presses create new(on pc) or signup here(on android) link. 4. The System respons with the login page.(Fig:3.1.1b)
3. Actor gives information to the labeled fields. Then he or she either presses login button or hits Enter. 4. System checks user inputes, saves new user information on database and redirects to the common dashboard.(Fig:3.1.1c)

5.Other Nonfunctional Requirements

5.1 Performance Requirements
  1. 95% of all response time will be less than 8.5 second.
  2. Specifying system workload which may specify high or even excessive volumes.
5.2 Safety Requirements
  1. On complex systems involving many interactions safety-critical functionality will be Identified and thoroughly analyzed before deriving hazards and design safeguards for mitigations.
  2. The number and complexity of safety critical interfaces will be minimized.
  3. Procedures Reliance on administrative for hazard control will be minimized.
  4. Contributing factors and root causes of faults and resultant hazards associated with the system and its software are identified, evaluated and eliminated or the risk reduced to an acceptable level, throughout the lifecycle.
  5. All data from outside program or service will be validated.
  6. Access Control—Deny by Default:Using server-side trusted data (data that has been properly validated) to make access control decisions.Checking all functions to make sure that the user is authorized before proceeding.
  7. Protecting Data and Privacy:Protecting data and privacy is about access control (which we’ve already talked about), auditing (which we’ll cover under logging), and encryption: encrypting data in transit, at rest, and during processing.
5.3 Security Requirement
  1. User will add some information in signup page.
  2. User will give email ID for signup and log in for the next time.
  3. User will give password of that email ID.
  4. User can read documents given by admin.
  5. Admin will keep record of all login history of each user.
  6. User won’t able to download the documents given by admin.
  7. Admin will be able to access how far the user read the given documents.
  8. Admin will control the privacy of uploading documents or posts.
  9. User can change their password.
  10. User won’t be able to download given documents.
5.4 Software Quality Attributes
  1. Adaptibility:This software will be adaptable in android,google crome browser or any.
  2. Correctness: The program code of this software is specific.The system of the software is not complex.
  3. Reliability: The inputs of the software are free of erros.
  4. Maintainability:The software is suitable for debugging (localization and correction of errors) and for modification and extension of functionality.
  5. Portability:It is hardware independent,extended of exploitation of specialized system function, system-dependent elements are collected in easily interchangeable program components.
  6. Testability:It is modular,it has structured programs suitable for systematic, stepwise testing.
  7. Robustness: Less frequent errors (e.g. power failure).
  8. Flexibility:It has a system to respond to potential internal or external changes affecting its value delivery, in a timely and cost-effective manner.
  9. Usability: The software will be used by specified users.
  10. Reusability:The software will be reusable.
  11. Interoperability: Its interfaces are completely understood, to work with other products or systems, at present or in the future, in either implementation or access, without any restrictions.
  12. Availability:As the availability depends on relaibility and testability.So the software will be available.
